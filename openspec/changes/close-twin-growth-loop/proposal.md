## 为什么

当前分身在冷启动后已经能基于 `active_version_id` 对话，但后续日常聊天里确认过的事实、经历和高代表性表达，还没有稳定回流成新的分身成长材料。结果是分身更像“带着初始快照和最近 transcript 的一次性角色”，而不是“会持续长的数字分身”。

这是分身智能化路线里最早必须补上的闭环。只有先让分身从真实日常里受控成长，后面的混合检索、人物地点图谱和叙事层才有稳定的落点。

## 变更内容

- 为分身补上“成长闭环”：把日常主链路中已经确认的 facts / events 和高代表性表达，转成分身后续成长的正式素材。
- 定义分身成长的触发与回写方式，让分身通过受控的版本刷新继续变长，而不是每轮聊天都直接漂移。
- 区分“临时 live expression”与“可沉淀的长期表达证据”，避免最近几句 transcript 直接把分身说话风格带偏。
- 保持现有产品边界不变：未确认线索不能直接进入分身成长；一人一分身约束不变；客观记忆层与画像层分离不被打破。

## 功能 (Capabilities)

### 新增功能

无

### 修改功能

- `twin`: 分身在冷启动完成后，必须能够从已确认的日常记忆和代表性表达中持续成长，并通过受控版本刷新体现到后续对话中。

## 影响

- 受影响数据：`memories`、`memory_facts`、`memory_events`、`companion_profiles`、`twin_profiles`、`twin_versions`
- 受影响逻辑：分身版本组装、表达层构建、日常确认记忆向分身成长素材的回写路径
- 受影响 API / 服务端路径：分身对话读取路径、分身版本更新辅助逻辑、日常记忆确认后的后续处理
- 可能受影响页面：主页日常链路、分身对话页、后续任何展示“分身已更新”状态的界面
- 不改变记忆确认边界、不改变一人一分身边界，也不允许客观编辑直接污染画像层
