## 上下文

当前系统已经有 `memory_facts`、`memory_events`、`twin_versions` 和最近 transcript 叠加，但缺少一个统一的运行时检索编排器。结果是分身能“拿到一些记忆”，却很难在模糊提问、同义词、代词、省略和阶段性提问中表现出联想能力。

本 change 解决的是运行时记忆获取，不负责人物关系抽取本身、记忆生命周期治理本身，也不负责最终 UI 呈现。

## 目标 / 非目标

**目标：**

- 让分身和其他运行时消费面可以通过统一检索层获取“记忆包”。
- 让检索同时考虑语义、人物、地点、时间和稳定事实。
- 让每次召回结果具备排序依据、来源追溯和降级路径。
- 在不破坏既有确认边界的前提下增强“联想感”。

**非目标：**

- 不在此 change 中定义人物地点抽取策略细节。
- 不在此 change 中定义记忆可见性与生命周期策略本体。
- 不在此 change 中新增地图、关系图、传记等前台界面。

## 决策

### 决策 1：新增运行时“记忆包”而不是让调用方直接查表

运行时统一输出记忆包，内容包括：

- 最相关事件
- 最相关事实
- 相关人物与地点引用
- 来源 memory ids
- 路由命中说明
- 置信度与敏感性标记

这样能让分身、回忆录增强版和未来地图/传记共享同一上下文构建方式。

替代方案：

- 让每个页面自己查 `memory_facts` / `memory_events`
- 继续只读 `twin_versions` 快照

不采用替代方案，是因为它们无法统一排序、降级和分寸控制。

### 决策 2：语义召回必须基于记忆切片，而不是整段 transcript

整段 transcript 过长且主题混杂，不适合作为语义召回最小单元。应以可追溯的记忆切片作为 embedding 基础，再在结果中回链到原始 memory 和确认后的结构化资产。

替代方案：

- 直接给整段 transcript 建 embedding

不采用替代方案，是因为会让召回噪声过大，也不利于解释“为什么命中”。

### 决策 3：混合召回包含显式降级路径

正常路径包含：

- 语义召回
- 人物召回
- 地点召回
- 时间召回
- 稳定事实召回

如果向量路由不可用，则保留结构化与关键词降级路径，避免分身完全依赖单一路由。

## 风险 / 权衡

- [风险] 多路召回会增加排序复杂度。
  → 缓解措施：先输出有限记忆包并记录各路命中分数，便于观察与调参。

- [风险] 向量路由不可用时体验回落明显。
  → 缓解措施：要求结构化召回和关键词召回始终可工作，并显式定义降级行为。

- [风险] 召回过多会污染回复质量。
  → 缓解措施：记忆包只保留高分、可解释、可追溯结果，不直接把全量结果交给模型。

## Migration Plan

1. 增加记忆切片与检索编排层。
2. 在分身对话链路中切换到记忆包输入。
3. 记录检索命中与降级情况，必要时保留旧快照路径作为回滚开关。
