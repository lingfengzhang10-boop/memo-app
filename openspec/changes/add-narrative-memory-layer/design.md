## 上下文

客观层已经能承接 facts / events，但用户真正想要的产品不只是知道“发生了什么”，还要知道“这件事在这个人的人生里意味着什么”。叙事层缺席时，分身能回答事实，却很难真正像这个人理解自己。

叙事层比事实层更主观，因此不能直接并入 facts / events，也不能让它污染客观层。

## 目标 / 非目标

**目标：**

- 把叙事记忆作为独立资产与客观层并存。
- 支持同一主题的多版本叙事与重新理解。
- 支持长期意义标签，例如转折点、反复模式、未解心结和人生主题。
- 让分身和回忆录能区分“发生了什么”和“如何理解它”。

**非目标：**

- 不在此 change 中定义所有意义标签的最终全集。
- 不在此 change 中替代事实、事件或人物地点层。
- 不在此 change 中实现传记或地图最终页面。

## 决策

### 决策 1：叙事资产与客观资产彻底分层

叙事层记录“解释、重述、理解”，不承担客观真值职责。事实仍在 facts / events 中保存，叙事通过链接指向它们。

### 决策 2：叙事允许多版本并存

用户对同一件事的理解可能随着时间改变，因此叙事层必须允许多个版本存在，并记录时间和来源，而不是以新叙事覆盖旧叙事。

### 决策 3：意义标签采用保守提炼策略

第一阶段只支持高价值、低歧义的意义标签：

- turning points
- recurring patterns
- life themes
- unresolved knots

AI 抽取失败时允许回退为“只保存叙事文本，不保存意义标签”。

## 风险 / 权衡

- [风险] 叙事层过度推断会显得像在替用户解释人生。
  → 缓解措施：要求保守抽取，并允许用户确认、纠正或降级为普通叙事记录。

- [风险] 叙事与事实混淆会污染回忆录和分身回复。
  → 缓解措施：所有消费面必须显式区分“事实引用”和“叙事理解”。

- [风险] 多版本叙事会增加检索复杂度。
  → 缓解措施：运行时优先使用最近确认版本，同时保留历史版本可回看。

## Migration Plan

1. 增加叙事层与意义标签模型。
2. 将后台理解链路扩展为叙事资产候选。
3. 让分身和回忆录逐步消费叙事层。

## Open Questions

- 叙事资产是否要求全部用户确认，还是允许部分仅保守存档。
- 意义标签是否与传记输出共用同一命名体系。
