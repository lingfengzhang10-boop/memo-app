## 上下文

现有 schema 已经有 `people`、`person_relationships` 和 `memory_person_links` 的基础，但产品和运行时链路尚未真正消费它们。地点目前主要以 `location_name` 等轻量字段存在，尚未成为独立的长期理解入口。

如果不把人物和地点正式做成长期资产，后续混合召回、关系图和人生地图都只能依赖散落字段，难以形成高质量理解。

## 目标 / 非目标

**目标：**

- 将人物和地点提升为正式长期资产。
- 允许人物别名、关系状态、地点归一化与来源记忆链接并存。
- 把人物/地点候选纳入确认流，而不是后台直接写入正式资产。
- 为分身和后续可视化能力提供稳定实体入口。

**非目标：**

- 不在此 change 中设计完整地理坐标系统或 GIS 能力。
- 不在此 change 中解决混合检索的总体排序问题。
- 不在此 change 中完成关系图或地图前台页面。

## 决策

### 决策 1：人物与地点都走候选确认流

人物和地点资产与 facts / events 一样，会显著影响长期理解，因此必须在入库前进入候选确认流。

替代方案：

- 后台直接抽取并写入人物、地点实体

不采用替代方案，是因为错误的人物归并和地点归并会长期污染整个系统。

### 决策 2：人物与地点都使用“规范名 + 别名”模型

人物与地点必须同时保留规范名和别名，以便处理“妈妈 / 我妈 / 母亲”“老家 / 河南老家 / 某个县城”等真实表达。

### 决策 3：地点先做语义归一，不强求精确坐标

第一阶段只要求地点能支持阶段理解、地图聚类和检索，不强制要求每个地点都有地理坐标。

## 风险 / 权衡

- [风险] 人物别名归并错误会影响所有后续检索。
  → 缓解措施：必须进入候选确认流，并保留来源记忆可追溯。

- [风险] 地点过早做成复杂地理系统会拖慢主线。
  → 缓解措施：先做语义地点实体，再评估是否引入坐标。

- [风险] 人物和关系抽取过度主观。
  → 缓解措施：先记录关系类型、状态和来源，不要求第一阶段自动推断复杂心理含义。

## Migration Plan

1. 完善人物与地点实体模型。
2. 将人物/地点候选接入确认流。
3. 在分身理解与后续可视化模块中消费已确认实体。

## Open Questions

- 地点是否独立为 `places` 表，还是先以内嵌规范化结构存在。
- 人物关系是否需要区分“用户与人物”关系和“人物与人物”关系。
