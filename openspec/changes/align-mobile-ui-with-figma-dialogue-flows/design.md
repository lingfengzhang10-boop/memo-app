## 上下文

当前移动端 Web 里实际存在两套对话入口：`/mobile` 上的 Liva 陪伴首页，以及 `/mobile/twin/[id]` 上的分身聊天页；它们又被 `/mobile/twin`、`/mobile/me`、`/mobile/review` 和底部导航分散开。Figma 这次提供的是另一套信息架构：用户进入后先看到单一的对话页，顶部是当前人物，左上角进入“更多”，点击人物头部进入“选择人物”，语音/文字只是同一条对话链路里的两种输入形态。

这意味着本次改动不只是视觉换皮，而是要把“Liva 对话”和“分身对话”收敛到同一个移动端对话壳里，同时保留现有两条后端链路的真实边界：Liva 路径继续走记忆沉淀、线索确认和画像聚合；分身路径继续走 active version、topic interaction 和专属语音。除此之外，用户还明确要求显示“其他用户授权给当前用户可以对话的人物”，而仓库里当前只有 `twin_profiles.share_enabled`，并没有授权映射表，所以这次必须补一层最小的授权读取模型。

用户随后修正了一个关键约束：人物选择页需要允许选择当前用户自己的分身，否则从分身详情发起对话后会回到统一首页并显示 Liva，破坏用户对“正在和自己的分身对话”的理解。同时，被授权人物仍必须带有创建分身时上传的独立照片素材，不能使用统一占位形象。这使得本变更不仅要重组路由，还要把分身照片作为正式产品资产纳入创建、存储和读取链路。

## 目标 / 非目标

**目标：**
- 让移动端主入口对齐 Figma，以 `/mobile` 为中心呈现文字/语音双模式对话页。
- 收敛 Liva、自己的分身与已授权人物到同一套“人物选择 -> 对话”的流转里；“更多 -> 分身详情 -> 对话”仍可作为进入自己分身的快捷入口。
- 让“更多”页能够根据用户是否已创建自己的分身，分别进入“创建分身”或“分身详情”分支。
- 在分身创建流程中加入照片上传，并确保该照片能作为授权人物卡片和分身详情页的独立形象素材。
- 在不破坏现有记忆确认、分身 active version、语音克隆边界的前提下，复用现有链路完成 UI 改版。
- 为本地图片/视频选择、用户信息补全和 Figma 中未细化的其他操作保留明确占位，而不是做成死链。

**非目标：**
- 不在本变更中实现图片/视频上传、媒体理解、文件预览或与对话联动的多模态链路。
- 不在本变更中实现“授权管理后台”或完整的共享分身运营能力；只实现读取显式授权结果所需的最小模型。
- 不改造 Web 工作台、回忆录编辑器、桌面端 `/twin` 系列页面的信息架构。
- 不变更 bootstrap 问题集、记忆表结构语义或分身成长算法。

## 决策

### 决策 1：移动端改为“统一对话壳 + 菜单驱动次级页”，不再以底部导航作为主入口
- 选择：将 `/mobile` 作为统一对话页，默认展示文字模式，通过显式切换进入语音模式；新增 `/mobile/characters`、`/mobile/more`、`/mobile/profile`、`/mobile/twin/[id]` 这类二级页承接 Figma 中的页面。
- 原因：Figma 的交互核心不是四个并列 tab，而是“对话页是中心，其余页面围绕对话页展开”。继续保留底部导航会把实现逼回当前信息架构，导致视觉改了但页面关系没改。
- 备选方案：
  - 继续保留 `MobileBottomNav`，只在原页面内换样式：视觉能更快上线，但会继续把 Liva、分身、我的拆在不同入口里，放弃。
  - 把文字页和语音页做成两个完全独立路由：会重复对话壳状态和跳转逻辑，放弃。

### 决策 2：使用统一的 `DialogueCharacter` 抽象承接 Liva、自己的分身与已授权人物
- 选择：新增一层移动端可用的人物模型，例如 `DialogueCharacter = { id, kind, title, subtitle, avatar, twinId?, isSystem, isShared }`，人物列表由三部分组成：
  - 固定的系统人物 `liva`
  - 当前用户自己的分身
  - 通过显式授权映射可被当前用户对话的其他分身
- 原因：顶部人物选择是用户理解“当前正在和谁说话”的主入口。如果自己的分身不能在这里被选中，用户从分身详情进入后会回落到 Liva 首页，造成错误心智。
- 备选方案：
  - 继续把 Liva 和已授权人物当成两套路由：无法实现一个统一的人物选择页，放弃。
  - 只在本地写死 Liva 和假数据卡片：能做演示，但无法支撑真实切换，放弃。

### 决策 3：补一个最小的授权读取模型，而不是滥用 `share_enabled`
- 选择：新增或等价提供一张授权映射表，例如 `twin_dialogue_grants`，至少包含 `twin_id`、`owner_user_id`、`grantee_user_id`、`status`、`created_at`，并配套 RLS，让当前用户只能看到被授权给自己的分身。
- 原因：`share_enabled` 只能表达“这个分身允许分享”，不能表达“分享给了谁”。用户这次要的是“其他用户授权给当前用户可以对话的人物”，这是具体授权对象，不是全局开关。
- 备选方案：
  - 继续只看 `share_enabled = true`：会把所有开启分享的分身暴露给任意登录用户，违反授权边界，放弃。
  - 暂时完全不做授权数据，只做 UI 占位：人物选择页无法进入真实对话，和用户要求不符，放弃。

### 决策 4：分身照片必须成为正式资产，并在创建时上传
- 选择：为分身增加独立照片素材字段或关联资产，创建分身时必须完成照片上传；人物选择页和分身详情页统一读取该资产，不使用共享占位图替代已授权人物。
- 原因：用户已经明确“授权形象必须得有”，且照片是在创建分身时上传。若不把它纳入正式资产链路，人物选择页只能退化为临时占位图，无法满足这次需求。
- 备选方案：
  - 使用默认头像或统一底图：和确认要求冲突，放弃。
  - 等未来做完整共享系统时再补照片：会导致这次 UI 无法真实落地，放弃。

### 决策 5：Liva 与分身继续复用各自后端链路，但在统一对话壳里共享同一种输入/回显结构
- 选择：
  - 当当前人物是 `liva` 时，沿用现有 `requestReflection` / clue confirmation / `memories` 写入链路。
  - 当当前人物是分身时，沿用现有 `/api/twin/chat`、topic interaction、voice clone 语音回复链路。
  - 两条链路都输出到统一的 UI 结构：顶部人物回复文本、底部用户文字回显、中部人物展示区、下方输入操作区。
- 原因：这次改动是 UI 收束，不应该把两条已经分化的业务链路再强行混成一个接口，否则会破坏当前验证过的行为。
- 备选方案：
  - 强行抽成一个统一后端接口：短期工作量大，且风险集中在业务回归，放弃。
  - 保留两套完全独立页面，只用视觉模拟统一：切换人物和回跳会非常别扭，放弃。

### 决策 6：未实现功能必须显示为“预留态”，不能消失或假装可用
- 选择：附件加号、本地图片/视频、分身详情里的其他操作、用户信息里的个人资料补全，以及 Figma 中有框但需求未细化的区域，都以明确的“预留”状态呈现，可点击进入占位弹层或占位单元，但不触发真实上传和保存。
- 原因：用户已经明确要求“其他没提到的先预留”。这意味着 UI 上要保留位置，技术上要阻止它误入真实链路。
- 备选方案：
  - 先把这些入口隐藏：实现简单，但和 Figma 对不上，放弃。
  - 入口可点但静默无反应：体验最差，放弃。

### 决策 7：线索确认边界继续存在，并以内嵌浮层或局部面板适配新对话页
- 选择：Liva 路径里已有的 fact / event 待确认线索，继续在移动端对话流中出现，但样式改为适配新对话页的浮层或局部卡片，而不是依赖旧首页结构。
- 原因：UI 改版不能以牺牲“先确认再入库”为代价；Figma 没有完整画出该状态，不代表可以删掉。
- 备选方案：
  - 把线索确认完全移到别的页面：会打断主链路，放弃。
  - 临时关闭线索确认：直接破坏产品边界，放弃。

## 数据流 / 确认流 / 回写流

### 数据流
```text
/mobile?character=<id>&mode=text|voice
  -> 加载 DialogueCharacter 列表
  -> 解析当前人物（Liva / 已授权人物）
  -> 渲染顶部回复文本、中部人物展示、底部输入操作

文字输入
  -> Liva: 直接走 reflection 链路
  -> 分身: 直接走 twin chat 链路

语音输入
  -> VoiceRecorder / ASR
  -> 得到 transcript 后进入与文字相同的人物分支
```

### 确认流
```text
Liva 对话（文字或语音）
  -> 即时回复
  -> 抽取 fact / event 候选
  -> 在新对话页内以浮层/卡片形式等待用户确认
  -> 仅 confirmed 线索写入 memory_facts / memory_events

分身对话
  -> 只读取 active version / topic context / voice clone
  -> 本次不新增“把分身对话直接沉淀为正式记忆”的行为
```

### 回写流
```text
Liva 路径
  -> memories
  -> companion_profiles
  -> memory_facts / memory_events（仅确认后）

分身路径
  -> twin topic interactions
  -> twin speech / voice replay 状态
  -> 继续读取 twin_profiles / twin_versions，不改变“一人一分身”边界

人物列表路径
  -> 读取系统 Liva 配置
  -> 读取当前用户自己的分身
  -> 读取 twin_dialogue_grants 解析已授权人物
  -> 读取对应 twin 的 portrait asset

分身创建路径
  -> bootstrap 问答
  -> 上传分身照片
  -> 持久化 twin portrait asset
  -> 后续被授权时可用于人物卡片和详情展示
```

## 风险 / 权衡

- [统一对话壳会让 `app/page.tsx` 和 `app/twin/[id]/page.tsx` 的职责重新切分] -> 先抽出人物列表与角色分支层，再把现有 Liva / 分身提交逻辑分别挂接进去，避免一次性重写业务函数。
- [新增授权映射表会引入新的 RLS 与数据可见性风险] -> 授权模型只做读取最小集，强制要求 owner / grantee 双边界校验，不在本变更中同时做复杂授权管理 UI。
- [分身照片成为正式资产后，会影响 bootstrap 流程和存储约束] -> 在本变更里只引入单图上传与读取，不扩展相册、多裁剪版本或复杂素材管理。
- [Figma 只明确了主要页面，没有给出所有状态页] -> 对附件、用户信息扩展、分身详情额外操作统一采用预留态，避免实现时凭空扩展。
- [去掉底部导航后，旧 `/mobile/me`、`/mobile/review` 的入口消失] -> 先从主导航中移除，但保留兼容路由或重定向，直到新的 Figma 页面接管它们。
- [语音/文字双模式都要显示用户文本回显，容易让状态管理变复杂] -> 将“最新用户话语”和“当前人物回复文本”抽成统一展示状态，不让每种输入模式各自维护一套 UI。

## Migration Plan

1. 先补人物列表抽象和读取来源，包括系统 Liva、自己的分身、已授权人物，以及分身照片资产读取。
2. 将分身创建流程补上照片上传与持久化，确保分身详情和授权人物卡片有真实素材来源。
3. 新建 Figma 对齐的移动端路由与页面壳，隐藏旧底部导航主入口，但保留兼容路由。
4. 将现有 Liva 对话链路和分身对话链路挂到统一对话页里，打通文字/语音双模式。
5. 接入“更多”页分支、分身详情和用户信息页，保留未实现操作的占位态。
6. 逐步替换旧的移动端入口与跳转；若回滚，只需恢复原 `app/mobile/layout.tsx` 导航与原页面导出，新加授权表与照片资产字段属于向前兼容的增量结构。
