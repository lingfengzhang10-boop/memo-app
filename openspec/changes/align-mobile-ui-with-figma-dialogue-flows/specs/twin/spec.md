## ADDED Requirements

### 需求:移动端人物列表必须显示当前用户可对话的分身人物
系统必须在移动端人物选择页中显示当前用户自己的分身，以及显式授权给当前用户可对话的其他分身人物；未授权给当前用户的其他分身不得出现。

#### 场景:用户查看可对话分身人物
- **当** 用户打开移动端人物选择页
- **那么** 系统必须显示当前用户自己的分身（如果已创建）
- **那么** 系统必须显示已显式授权给当前用户可对话的其他分身人物
- **那么** 系统不得显示未授权给当前用户的其他分身

#### 场景:用户从分身详情发起对话
- **当** 用户在移动端分身详情页点击“与分身对话”
- **那么** 系统必须回到统一对话页并把该分身设为当前对话人物
- **那么** 后续对话必须继续读取该分身的 active version

#### 场景:用户从人物选择页选择自己的分身
- **当** 用户在移动端人物选择页点击自己的分身
- **那么** 系统必须回到统一对话页并把该分身设为当前对话人物
- **那么** 页面顶部必须显示该分身，而不是 Liva
- **那么** 后续对话必须继续读取该分身的 active version

### 需求:分身创建时必须上传可展示的独立照片素材
系统必须在分身创建流程中采集并持久化该分身的独立照片素材，以供分身详情页和后续被授权对话时的人物卡片展示使用。

#### 场景:用户创建自己的分身
- **当** 用户完成创建分身流程
- **那么** 系统必须要求用户上传该分身的独立照片素材
- **那么** 系统必须持久化该素材并与该分身建立稳定关联

#### 场景:授权分身被其他用户看到
- **当** 某个分身被授权给其他用户对话
- **那么** 系统必须向被授权用户展示该分身已上传的独立照片素材
- **那么** 系统不得为该分身回退到统一占位头像

## MODIFIED Requirements

### 需求:Shared twin is a future extension, not default behavior
分身对外对话必须以显式授权名单为前提，而不是默认对所有用户开放；系统可以在移动端人物列表中展示已授权的外部分身，但必须始终把它们表达为“被授权可对话的人物”，而不是默认等同于其本人。

#### 场景:当前用户看到外部分身人物
- **当** 当前用户在移动端看到某个不属于自己的分身人物
- **那么** 该人物必须来自显式授权结果
- **那么** 系统必须只允许当前用户在授权范围内与其对话

#### 场景:产品展示外部分身人物
- **当** 系统在人物选择页、更多页或分身详情页展示外部分身
- **那么** 系统必须将其表述为被授权可对话的人物或分身
- **那么** 系统不得暗示该人物默认完全等同于其本人
