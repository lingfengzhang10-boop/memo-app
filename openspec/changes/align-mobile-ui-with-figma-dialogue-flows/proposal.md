## 为什么

当前移动端入口仍然分散在首页、分身列表、分身聊天、我的和底部导航之间，和 Figma 中“以单一对话页为中心”的信息架构不一致。现在需要先把移动端 UI 收束到 Figma 定义的对话流，并补齐最小的人物授权列表、分身详情入口和分身照片素材，后续实现才不会继续叠加在旧页面结构上。

## 变更内容

- 将移动端从“底部导航 + 分散页面”调整为“统一对话页 + 菜单驱动二级页”的结构，对齐 Figma 中的 `对话-文字`、`对话-语音`、`选择人物`、`更多`、`更多-未创建分身`、`分身详情`、`用户信息`。
- 让 `/mobile` 默认进入文字对话页，并在同一条对话链路内切换到语音对话页；两种模式都保留顶部人物回复文本和底部用户文本回显。
- 新增人物选择页，整合系统默认人物 Liva、当前用户自己的分身，以及显式授权给当前用户可对话的其他人物；用户从顶部人物选择切到自己的分身后，必须留在统一对话页并走分身对话链路。
- 将“更多”页拆成“未创建分身”和“已有分身”两种状态；已有分身时可进入分身详情并发起对话，未创建分身时复用现有 bootstrap 问答页创建分身，并要求在创建流程中上传分身照片。
- 预留附件选择、本地图片/视频入口、用户信息扩展位和 Figma 中未细化的其他操作，但本变更不实现完整上传、媒体处理或个人资料编辑。
- 保留现有记忆确认、分身 active version、分身语音克隆等后端链路，只补充支撑新 UI 所需的最小功能增量，包括“授权人物列表”的读取能力，以及分身照片素材的上传、持久化与展示能力。

## 功能 (Capabilities)

### 新增功能
- `mobile-dialogue-surfaces`: 定义移动端 Figma 对话流、人物选择、更多/详情/用户信息页面，以及未实现操作的预留态行为。

### 修改功能
- `memory-companion`: 将日常陪伴入口改为统一对话页中的文字/语音双模式，而不是当前单一沉浸式语音首页。
- `twin`: 将分身入口改为“人物列表 + 更多页 + 分身详情”的结构，并支持显式授权的可对话人物出现在移动端人物列表中，同时要求分身创建时上传并持久化独立照片素材。

## 影响

- 受影响的页面与路由：`app/page.tsx`、`app/mobile/layout.tsx`、`app/mobile/page.tsx`、`app/mobile/twin/*`、新增的移动端人物选择/更多/用户信息页面。
- 受影响的组件与样式：`components/VoiceRecorder.tsx`、`components/MobileBottomNav.tsx`、`app/globals.css`，以及新的移动端对话/人物卡片组件。
- 受影响的客户端逻辑：`lib/clientSurfaceRouting.ts`、现有 Liva 对话链路、现有分身对话链路、分身资料读取逻辑。
- 受影响的数据与权限：需要新增或等价提供“某个分身被授权给当前用户对话”的读取来源；现有 `share_enabled` 仍不足以表达具体授权对象；同时需要为分身补充可展示的照片素材字段或关联资产来源。
- 保持不变的边界：facts / events 仍然先确认再入库；画像层仍不被客观编辑直接污染；每个用户仍然只能拥有一个自己的分身。
