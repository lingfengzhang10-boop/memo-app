## ADDED Requirements

### 需求: Native App 必须作为高频语音入口
系统必须将 Native App 定义为用户日常进入产品时的高频入口，并优先承载录音、即时回应、轻量确认、分身语音对话和提醒回流等强依赖系统能力的体验。

#### 场景: 用户在 App 中进行日常记录
- **当** 用户打开 Native App 并准备记录当下想法
- **那么** 系统必须优先提供直接的录音入口，而不是要求用户先进入深度编辑页面
- **那么** 系统必须允许用户在录音后立即看到即时回应和轻量记忆确认入口

#### 场景: 用户在 App 中继续与分身互动
- **当** 用户在 Native App 中进入分身相关界面
- **那么** 系统必须把分身对话、语音播放和必要设置视为高频入口能力
- **那么** 系统不得要求用户先进入 Web 工作台才能完成基础的分身日常使用

### 需求: Web 必须作为深度整理与资产查看工作台
系统必须将 Web 定义为深度整理、编辑、核对、运营和资产查看的主要工作台，并优先承载回忆录、记忆工坊、全量地图、关系图和传记视图等高信息密度能力。

#### 场景: 用户需要深度编辑记忆资产
- **当** 用户需要批量确认、修正、归档或核对记忆内容
- **那么** 系统必须在 Web 上提供完整的整理与编辑能力
- **那么** 系统不得要求首版 Native App 同时完整复制这些高密度工作台能力

#### 场景: 用户需要查看全量资产结构
- **当** 用户需要查看完整的人生地图、关系图、传记或调试信息
- **那么** 系统必须允许这些能力以 Web 为主要呈现面
- **那么** 首版 Native App 可以只提供轻量回顾，而不必完整承载这些重界面

### 需求: App 与 Web 必须共享同一后端契约与数据真相
系统必须保证 App 与 Web 对记忆确认、分身版本、音色配置和后续资产读取使用同一套后端契约，不得出现双端各自维护不同真相的情况。

#### 场景: 用户在 App 中确认线索后切换到 Web
- **当** 用户在 Native App 中确认了一条事实或经历线索
- **那么** Web 必须能够读取到同一条已确认的正式记忆记录
- **那么** 系统不得要求用户在 Web 中再次确认同一条线索

#### 场景: 用户在 Web 中整理资产后回到 App
- **当** 用户在 Web 中编辑或整理了可共享给分身的正式资产
- **那么** Native App 必须在后续读取中看到相同的确认结果、active version 和相关摘要
- **那么** 系统不得让 App 和 Web 对同一用户展示不同的分身真相

### 需求: 首版 App 信息架构必须保持克制
系统必须将首版 Native App 的主要信息架构限定为“念及、分身、回顾、我的”四个主页面，并将“分身冷启动”和“音色样本录制”作为独立流程处理。

#### 场景: 用户首次安装并进入 App
- **当** 用户首次进入 Native App
- **那么** 系统必须允许用户通过独立的分身冷启动流程完成初版分身创建
- **那么** 系统不得把完整回忆录、全量地图或全量关系图强行放入首版主导航

#### 场景: 用户配置分身音色
- **当** 用户需要录制或重录分身音色样本
- **那么** 系统必须提供独立的音色样本流程或设置入口
- **那么** 系统不得把音色样本配置与日常聊天主流程混为一体
