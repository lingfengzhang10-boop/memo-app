# define-app-web-surface-boundaries

## 1. 共享边界与契约
- [x] 1.1 盘点当前录音、转写、播放、确认、分身成长链路中哪些属于共享领域逻辑，哪些属于浏览器平台实现
- [x] 1.2 收口 App 与 Web 共用的后端契约，明确记忆确认、active version、音色配置和回忆录读取都复用同一套数据真相
- [x] 1.3 验证现有“一人一分身”“确认后入库”“画像层不被客观编辑污染”三条边界在双表面方案下仍然成立

## 2. 首版 App 表面
- [x] 2.1 搭建首版 App 的四个主页面信息架构：念及、分身、回顾、我的
- [x] 2.2 实现 App 侧“念及”首页的高频录音入口、即时回应和轻量线索确认闭环
- [x] 2.3 实现 App 侧“分身”页面的高频文字/语音对话入口，并接入共享的 active version 与音色配置
- [x] 2.4 实现 App 侧“回顾”和“我的”页面，只承载轻量回顾、能力状态和必要设置，不复制 Web 的重工作台
- [x] 2.5 实现分身冷启动与音色样本录制两个独立流程，并确保它们不混入日常聊天主界面

## 3. Web 工作台收口
- [x] 3.1 明确保留 Web 侧回忆录、记忆工坊、全量地图、关系图、传记和调试入口，不在首版 App 中重复实现
- [x] 3.2 调整现有 Web 页面职责，使其更偏向深度整理、编辑、核对和运营支持，而不是继续承担唯一高频入口角色
- [x] 3.3 验证用户在 Web 上完成的整理结果会继续通过共享契约回流到 App 可见的记忆与分身状态

## 4. 平台适配与失败兜底
- [x] 4.1 为录音、ASR、音频播放、通知和本地缓存建立平台适配边界，避免业务逻辑继续直接绑定浏览器 API
- [x] 4.2 为 App 与 Web 分别定义语音链路失败时的兜底路径，确保 ASR、TTS、上传失败时仍可继续文字流程或稍后重试
- [x] 4.3 完成跨端 QA：App 中确认的线索可在 Web 看到，Web 中整理的正式资产可在 App 读取，分身始终只共享一个 active truth
