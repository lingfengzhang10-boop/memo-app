## 上下文

当前产品已经在移动端 Web 上跑通了语音录入、服务端 ASR、即时回复、线索确认、回忆录和分身对话，但“客户端表面”与“产品能力”仍然绑定得过紧：录音、播放、权限、页面编排和记忆确认都堆在浏览器页面里。随着产品继续向分身成长、混合检索、人生地图和关系图推进，继续默认“Web 页面就是唯一载体”会同时拖慢两件事：一是高频语音体验无法充分利用系统能力，二是深度整理与资产查看会被迫适配移动端入口而失真。

利益相关者有三类：日常使用的用户、需要稳定语音链路与系统整合的移动端入口、以及需要整理与核对数字资产的深度工作台。这个设计不是要立即重写成原生 App，而是要先定清“App 负责什么、Web 保留什么、后端共享什么”的边界，让后续客户端实施有稳定的落点。

## 目标 / 非目标

**目标：**
- 定义 Native App 与 Web 的职责分工，避免未来两个客户端各自复制一套业务逻辑。
- 明确首版 App 的信息架构，只承载高频语音入口、分身对话、轻量回顾和必要设置。
- 保证记忆确认、分身成长、音色配置和未来检索能力在 App 与 Web 上共享同一数据与后端契约。
- 为后续原生能力接入预留平台适配层，尤其是录音、ASR、播放、推送、本地缓存等高依赖系统能力的部分。

**非目标：**
- 不在本 change 中决定具体原生技术栈（如 Expo、React Native 或纯原生）。
- 不要求首版 App 完整承载回忆录、人生地图、关系图或传记导出。
- 不改变现有记忆确认边界、一人一分身边界或画像层隔离边界。
- 不在本 change 中引入新的记忆表结构、检索算法或人物地点图谱实现。

## 决策

### 决策 1：采用“App 高频入口 + Web 深度工作台 + 共享后端”的双表面架构
- 选择：Native App 承担高频、语音优先、系统能力强依赖的使用场景；Web 承担深度整理、编辑、运营和资产查看；Supabase、Next.js API 与 AI 编排层继续作为共享中台。
- 原因：这个产品的高频入口越来越依赖录音、音频播放、权限、推送和后台恢复，而深度整理又更适合大屏和编辑工作流。把两种任务都塞进单一 Web 表面，会同时损失语音体验和整理效率。
- 备选方案：
  - 继续只做移动端 Web：验证快，但语音与系统能力上限过低，后续迁移成本更大，放弃。
  - 立即全量重写成原生 App：会在护城河能力尚未成熟时把主要精力转向客户端重做，放弃。

### 决策 2：首版 App 只承载 4 个主页面和 2 个独立流程
- 选择：首版 App 只包含“念及”“分身”“回顾”“我的”四个主页面，以及“分身冷启动”“音色样本录制”两个独立流程。
- 原因：首版 App 的目标是验证高频回访和语音体验，而不是复制 Web 工作台。页面数量越多，越容易把 App 做成一个稀释了主动作的信息系统。
- 备选方案：
  - 直接把回忆录、地图、关系图全部搬进首版 App：功能完整但会拖慢上线，也会削弱语音入口的清晰度，放弃。
  - 只做一个聊天页：太窄，无法验证回顾、提醒和分身设置的回流价值，放弃。

### 决策 3：Web 继续承担“整理与资产消费”的重界面
- 选择：Web 保留回忆录编辑、记忆工坊、全量人生地图、关系图、传记视图、数据核验和运营支持能力。
- 原因：这些界面天然需要更强的信息密度、编辑空间和调试视角，不适合作为首版 App 的主任务。
- 备选方案：
  - 完全弱化 Web，只把它当营销页：会丢失当前最成熟的编辑与调试阵地，放弃。
  - 让 App 和 Web 同时维护全量编辑能力：会导致双端逻辑分叉，放弃。

### 决策 4：客户端表面变化不能改变共享数据与确认契约
- 选择：无论入口来自 App 还是 Web，录音、转写、AI 提取、线索确认、结构化记忆入库、分身成长和 active version 读取都必须复用同一套后端契约。
- 原因：产品真正的资产是记忆、关系和分身版本，而不是某个客户端表面。双端如果出现不同的确认规则或不同的 active version 解释，会直接破坏记忆可信度。
- 备选方案：
  - App 维护独立的轻量数据模型，之后再同步到 Web：会制造双重真相，放弃。
  - App 完全只做壳，不暴露任何记忆确认：会让高频入口断在最关键的沉淀一步，放弃。

### 决策 5：原生能力以平台适配层承载，业务逻辑不直接绑死在浏览器 API 上
- 选择：未来需要把录音、ASR 路由、音频播放、通知、本地缓存等收口成平台适配层，由 Web 与 Native 分别实现，记忆确认和分身成长逻辑保持在共享领域层与后端层。
- 原因：当前代码已经明显依赖浏览器能力；如果不先抽离职责，未来原生客户端只能复制页面逻辑，而不是复用产品能力。
- 备选方案：
  - 继续让每个页面自己处理平台能力：短期快，长期无法维护，放弃。
  - 把所有能力直接搬到后端：无法解决录音、播放、系统权限这些终端问题，放弃。

### 决策 6：App 与 Web 都必须保留失败兜底，但兜底方式可以不同
- 选择：App 优先走系统或原生能力更强的链路，但当 ASR、TTS、上传、通知失败时，必须回退到“文字可继续、确认稍后补、分身仍可聊天”的模式；Web 继续保留现有服务端链路和轻量本地实验能力。
- 原因：语音链路失败不能阻断记忆沉淀和分身关系，这一点在 App 与 Web 上都必须成立，但具体兜底方式应服从各自平台能力。
- 备选方案：
  - 把所有能力都强绑到云端：一旦余额或网络出问题，整条链路瘫痪，放弃。
  - 只依赖端侧能力：一致性与可控性不足，且无法覆盖所有设备，放弃。

## 数据流 / 确认流 / 回写流

### 数据流
```text
App 录音 / Web 录音
  -> 共享上传与转写契约
  -> AI 即时回应 / 线索提取 / 画像聚合
  -> confirmed facts/events 写入结构化记忆层
  -> 分身成长逻辑刷新 active version
  -> App 与 Web 读取同一 twin profile / twin version / memoir assets
```

### 确认流
```text
AI 提取 fact / event 候选
  -> App 轻量确认卡片 或 Web 记忆工坊确认
  -> 只有 confirmed 线索进入正式 memory_facts / memory_events
  -> 未确认线索禁止直接污染 active version 或画像层
```

### 回写流
```text
App 高入口产生的新录音 / 轻量确认
  -> 共享后端写入 memories / facts / events / profiles
  -> twin growth 刷新 active version

Web 深度整理产生的编辑 / 归档 / 核对
  -> 共享后端写入 objective memory 或编排层
  -> 仍需遵守画像层不被直接污染的边界
```

## 风险 / 权衡

- [首版 App 范围过大，导致上线速度和验证效率下降] -> 把 App 范围限定为 4 个主页面和 2 个独立流程，资产化重界面继续留在 Web。
- [双端并存后，确认规则或 active version 解释出现分叉] -> 约束所有确认、成长和读取都复用共享后端与领域契约，不允许客户端自定义真相。
- [原生能力路线未定，过早绑定技术栈] -> 本 change 只定义产品边界与能力归属，不把具体框架写死。
- [App 优先后，Web 入口被忽视，影响当前验证速度] -> 保留 Web 作为现阶段主验证面和深度工作台，而不是立即替换。
- [ASR/TTS 等 AI 链路仍可能失败] -> 规定 App 与 Web 都必须支持文字继续、稍后重试和可回落的兜底，不让 AI 失败阻断主流程。

## Migration Plan

- 先保持当前 Web 主链路继续验证产品，避免在本阶段重写整个客户端。
- 在后续客户端实施时，以本 change 作为边界约束：先实现首版 App 的高频入口，不迁移 Web 的重编辑页面。
- 随着 App 落地，再逐步把录音、播放、提醒、缓存等能力从浏览器实现抽离到平台适配层。
- 如果 App 侧某项原生能力不稳定，可临时保留 Web 入口和现有服务端链路，不影响共享数据层与分身成长层。

## Open Questions

- 首版 App 是否需要在“回顾”页直接显示轻量记忆确认，还是统一回到首页卡片流完成确认？
- 分身页是否要在首版 App 就暴露“这条回复来自哪些记忆”的解释层，还是先只验证对话与音频体验？
- Web 的记忆工坊是否要在 App 首版同步提供只读入口，还是完全留在 Web？
- 未来原生客户端的技术栈是否优先围绕跨平台方案，还是先以单平台验证语音体验？
