## 为什么

当前产品已经能在移动端 Web 上跑通语音录入、记忆确认、回忆录和分身对话，但客户端职责仍然混在一起：高频语音入口、系统能力整合、深度编辑和资产查看都堆在同一套 Web 页面里。现在需要先把 App 与 Web 的产品边界写清楚，避免后续在做原生客户端时重走一遍信息架构，也避免未来的地图、关系图、回忆录和分身能力继续被“浏览器页面”牵着走。

## 变更内容

- 明确产品采用“双表面”策略：Native App 负责高频、语音优先、系统能力强依赖的入口体验；Web 负责深度整理、编辑、运营和资产查看。
- 定义首版 App 的核心范围：念及首页录音、分身聊天、轻量回顾、我的/设置，以及分身冷启动和音色样本两个独立流程。
- 定义 Web 的持续职责：回忆录编辑、记忆工坊、全量人生地图、关系图、传记视图、数据调试与运营支持。
- 明确共享中台边界：Supabase 数据模型、Next.js API、AI 编排、记忆确认、分身成长和未来混合检索逻辑继续作为 Web 与 App 共用的后端与领域层。
- 明确平台边界不能突破既有产品约束：客观线索仍需确认后入库，一人一分身仍保留，画像层仍不能被客观编辑直接污染。

## 功能 (Capabilities)

### 新增功能
- `client-surfaces`: 定义 Native App 与 Web 的职责划分、首版 App 信息架构，以及共享后端与领域层的边界。

### 修改功能
- `memory-companion`: 将日常记忆入口从“主页页面”扩展为“App 优先的语音入口 + Web 可保留的辅助入口”，并明确记忆确认流在双表面之间的一致行为。
- `twin`: 将分身能力拆分为 App 优先的高频对话入口与 Web 可保留的管理/观察入口，并要求共享同一 active version、音色配置和成长结果。
- `memoir`: 将回忆录明确为 Web 优先的深度整理与资产查看界面，而不是首版 App 必须完整承载的主界面。

## 影响

- 受影响的规范：`memory-companion`、`twin`、`memoir`
- 新增规范：`client-surfaces`
- 受影响的产品边界：记忆确认流、一人一分身边界、画像层不被客观编辑污染的边界均保持不变，但需要在 App 与 Web 上显式复用
- 受影响的代码与系统范围将包括现有 Web 页面、未来原生客户端信息架构、录音/播放/通知等平台适配层，以及与 `app/api/*`、Supabase 数据表和 AI 服务共用的后端契约
