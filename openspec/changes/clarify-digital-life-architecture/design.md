## 上下文

当前产品已经有原始录音与转写、事实与事件抽取、回忆录、分身冷启动与分身对话，但这些能力仍围绕“对话是否顺滑”组织。昨天确认的产品目标已经超出这个范围：系统不仅要回答得像本人，还要把人生内容沉淀为可检索、可追溯、可呈现、可继承的数字资产。

现状中最容易失控的地方有三个：

- 数据层已有 `memories`、`memory_facts`、`memory_events`、`companion_profiles`，但缺少统一的资产分层语义。
- 分身当前主要消费版本快照与最近 transcript，尚未被明确为“受治理的消费面”。
- 后续要补的能力很多，包括混合检索、人物地点关系、叙事层、资产化呈现，如果没有顶层设计，会重新把事实、叙事、表达和可见性混在一起。

## 目标 / 非目标

**目标：**

- 把产品主张明确成“数字生命资产系统”，而不是泛化聊天助手。
- 给后续 change 提供统一的信息分层、边界和推进顺序。
- 明确分身、回忆录、地图、关系图、传记都必须从同一套受治理资产读取。
- 保持既有关键边界不被破坏：记忆确认先于入库、画像层不被客观编辑污染、一人一分身。

**非目标：**

- 不在此 change 中直接设计数据库字段或 API 细节。
- 不在此 change 中直接引入向量检索、人物图谱、地图页面或 3D 动作。
- 不在此 change 中实现任何应用代码。

## 决策

### 决策 1：以后所有长期内容都按“资产层”而非“单表”思考

系统后续统一按七层资产理解：

- 原始层：录音、转写、原话片段
- 事实层：偏好、规则、习惯、状态
- 事件层：时间线与转折
- 人物层：人物实体与关系
- 地点层：地点实体与阶段场景
- 叙事层：如何理解自己的人生
- 表达层：如何说话、如何安慰、如何停顿

选择这一方向，是因为仅按表结构思考会让未来每次扩展都重新绕回 transcript 或 profile；按资产层思考则能让检索、分身、回忆录和传记共享同一语义基础。

替代方案：

- 继续沿用“memories + facts/events + profile”的轻量分层
- 仅在实现阶段逐步堆加新表

不采用替代方案，是因为这会导致后续新增人物、地点、叙事、意义层时不断回头改口径。

### 决策 2：分身被定义为“受治理的消费面”

分身以后必须消费经过筛选和治理的记忆包，而不是直接消费任意 transcript 或裸表数据。这样才能同时满足“更像本人”和“更有分寸”两个目标。

替代方案：

- 继续让分身主要依赖 `twin_versions` 快照和最近 transcript

不采用替代方案，是因为这只能支持“像在说话”，不能支持“像在理解这个人”。

### 决策 3：后续工作按模块 change 拆分，而不是继续做一个大一统 change

后续拆分为五个模块：

- 混合记忆检索
- 人物地点关系图谱
- 记忆治理
- 叙事与意义层
- 资产化呈现面

替代方案：

- 用一个总 change 包住所有后续工作

不采用替代方案，是因为这样会让实现阶段失去优先级，也不利于逐步上线和验证。

## 风险 / 权衡

- [风险] 总向 change 只写方向，不直接落地功能，容易被视为“只有文档”。
  → 缓解措施：同步拆出五个模块 change，并让它们各自达到 `/opsx:apply` 可实施状态。

- [风险] 顶层方向过宽，后续模块仍可能边界重叠。
  → 缓解措施：在各模块设计中明确输入输出、依赖关系和不负责内容。

- [风险] 新增层次太多，可能让产品显得复杂。
  → 缓解措施：方向层负责统一语言，真正的用户界面仍保持少而清晰，复杂性留在系统内部。

## Migration Plan

1. 先通过本 change 固化顶层产品与规范方向。
2. 立即用模块 change 承接具体能力设计，避免总向文档空转。
3. 后续实现按“治理先于召回、召回先于呈现”的顺序推进。

## Open Questions

- 叙事层和意义层是否在产品语言上分成两个概念，还是收在同一层中。
- 未来公开传记与私密自述是否共用一套资产，只靠可见性区分。
- 3D 联动是否作为数字生命呈现面的子模块，还是单独立项。
