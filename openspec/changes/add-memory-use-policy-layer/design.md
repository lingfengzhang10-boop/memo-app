## 上下文

当前系统已经实现了几层重要能力：

- `memories` / transcript 作为原始证据层。
- `memory_facts` / `memory_events` 作为确认后的结构化资产层。
- `semantic_memory_chunks` 作为语义召回底座。
- situational routing 负责把带地点、时间、人物锚点的问题收窄到局部语境。
- `twin_topic_interactions` 负责记录同一 asker 与同一 topic 的短期互动痕迹。
- prompt rebalance 已尝试让模型更自然地表达，而不是直接执行大量规则。

但测试暴露出一个新的系统缺口：这些能力回答的是“有什么记忆”和“当前召回到什么”，没有稳定回答“本轮应该怎么使用这些记忆”。当用户反复问“你在杭州过得怎么样”或追问“还有别的吗”时，系统仍可能反复复述同一事实，或者为了继续回答而套用不属于杭州语境的其他记忆。

这次变更引入记忆使用策略层，放在检索召回与模型生成之间，负责把“可召回材料”转换为“本轮可用记忆包”和“回答推进指令”。

## 目标 / 非目标

**目标：**

- 明确记忆从输入到回答的准入状态：原始记录、候选记忆、可信记忆、长期稳定记忆、只留档内容。
- 在分身聊天链路中增加记忆包裁剪，避免无关地点、人物、阶段或未确认事件进入本轮回答。
- 让同一话题重复提问时能够递进、换角度或自然收束，而不是机械复述。
- 让“还有别的吗”只能消费当前 topic 下未讲过且可信的细节，找不到时必须诚实收束。
- 增加程序性记忆规则，约束回答方式，而不是把这类规则混入人生事实层。
- 建立固定 QA 回归集，让后续每次调整都能验证“重复、越界、编造、收束”这些核心风险。

**非目标：**

- 不推翻现有 semantic substrate、situational routing、topic interaction memory。
- 不把未确认 raw evidence 直接提升为正式 `memory_facts` / `memory_events`。
- 不新增完整的用户可见记忆管理后台。
- 不在本次变更中做完整知识图谱、人生地图或传记生成。
- 不承诺完全解决所有自然语言幻觉，只先收敛高频分身对话失败场景。

## 决策

### 决策 1：新增运行时 `MemoryUsePolicy`，而不是继续堆 prompt

运行时在模型生成前构造一个策略对象，至少包含：

- `inputTrust`: 本轮用户输入可信度与是否适合沉淀。
- `topicScope`: 当前问题允许使用的时间、地点、人物、阶段、主题边界。
- `allowedMemoryItems`: 本轮允许进入 prompt 的事实、事件、原文证据。
- `blockedMemoryReasons`: 被排除的候选及原因，如跨主题、未确认、低置信、已讲过。
- `answerProgressionMode`: `fresh_answer` / `deepen_answer` / `diversify_answer` / `graceful_close` / `fuzzy_recall`。
- `proceduralRules`: 本轮表达约束，如不要原样重复、不要编新事件、无新增细节时收束。

替代方案是继续在 prompt 中写更多禁止规则。该方案不采用，因为 prompt 会越来越像规则执行器，而且模型仍可能在材料过多时抓错记忆。

### 决策 2：记忆准入状态服务于“能不能被用”，不替代确认流

输入链路保留原有确认边界，但新增准入状态语义：

```text
raw        原始记录，保留但不代表可信
candidate  候选记忆，需要确认或更多证据
confirmed  已确认，可参与正式回答
stable     长期稳定，可高优先级参与分身表达
archived   只留档，不主动参与分身回答
```

这些状态可以优先落在 metadata 或轻量字段中，首版不强制大规模迁移历史数据。已有 confirmed facts/events 默认可视为 confirmed，明确被污染或来源不稳的数据可以后续转为 archived。

### 决策 3：检索召回先变成“记忆包”，再进入 prompt

分身聊天不再把所有相关摘要并列喂给模型，而是先做记忆包裁剪：

```text
召回候选
  -> topicScope 过滤
  -> 准入状态过滤
  -> 已讲过角度过滤
  -> 当前问题意图过滤
  -> allowed memory packet
```

例如用户问“还有别的吗”时，策略层只能在当前杭州 topic 中寻找未讲过的可信细节。如果没有，输出 `graceful_close`，提示模型自然表达“现在能想起来最深的还是这些”，而不是抓其他主题补空。

### 决策 4：程序性记忆独立于人生事实

程序性记忆描述“分身怎么回答”，不描述“这个人经历了什么”。首版以代码常量或模板形式实现，未来可演进为配置表。

首版程序性规则包括：

- 禁止用未确认记忆创造新人物、新地点、新事件。
- 同一 topic 短时间重复时禁止复述上一轮主句。
- 追问“还有别的吗”时必须优先补充当前 topic 的未讲过细节。
- 找不到新增细节时必须自然收束，而不是编造或跨主题扩写。
- 不确定时可以表达“不太确定 / 现在想起来主要是这些”，但不要求用户额外确认。

### 决策 5：保留调试解释，但不把内部术语直接给模型

策略层需要输出 debug 信息，便于定位为什么某条记忆被允许或阻断。但进入 prompt 前要翻译成自然语言提示。

```text
debug: answerProgressionMode = graceful_close
prompt: 这段刚聊到这里，如果没有新细节，就简短自然地收住。
```

这样既保留工程可观测性，也避免模型复述系统术语。

### 决策 6：QA 回归先覆盖最容易破坏沉浸感的场景

首版 QA 以固定文本输入为主，避免 ASR 额度、浏览器麦克风和音色合成干扰。最小用例包括：

- 首问：“你在杭州过得怎么样”
- 立即重复：“你在杭州过得怎么样”
- 追问：“还有别的吗”
- 局部追问：“妈妈那时候做了什么”
- 否定式试探：“你在杭州有没有朋友聚会”
- 跨主题防污染：“你那时候是不是经常吃辣条”

这些用例必须验证：不原样重复、不跨主题抓取、不编造未确认事件、没有新增细节时自然收束。

## 风险 / 权衡

- [风险] 策略层过严会让分身变得保守。  
  缓解措施：优先使用 `deepen_answer` 和 `diversify_answer`，只有没有可信新增细节时才 `graceful_close`。

- [风险] 准入状态首版如果只靠 metadata，历史脏数据仍可能进入回答。  
  缓解措施：先在运行时对低置信、跨 topic、无来源、已污染模式做过滤；后续再补治理后台或清洗任务。

- [风险] 记忆包裁剪错误会漏掉真实细节。  
  缓解措施：debug 输出 blocked reasons，测试时能看到被排除原因；保留 feature flag 方便回滚到现有路径。

- [风险] 程序性规则过多会再次让 prompt 机械。  
  缓解措施：规则在服务端做裁剪和翻译，只给模型短提示，不把完整规则清单原样塞入 prompt。

- [风险] “无需用户确认”的沉浸要求可能与纠错需要冲突。  
  缓解措施：回答阶段不打断用户确认；记忆沉淀阶段仍保留候选/确认/归档状态，必要时在后续记忆管理入口处理。

## Migration Plan

1. 新增策略构建模块，先消费现有 facts/events/chunks/topic interactions，不立即要求迁移历史数据。
2. 在 `/api/twin/chat` 中插入策略层：召回后、prompt 构建前。
3. 将 prompt context 改为接收 allowed memory packet 与自然语言推进提示。
4. 为日常记忆写入补充准入状态与低置信过滤，默认不破坏现有确认流。
5. 增加固定 QA 脚本或测试说明，优先用文字输入验证。
6. 保留回滚开关；如果策略层异常，可临时回退到现有 situational routing + prompt 路径。
