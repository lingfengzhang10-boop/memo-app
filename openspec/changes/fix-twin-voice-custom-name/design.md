## 上下文

当前分身音色克隆由 `app/api/twin/voice/clone/route.ts` 代理到上游语音克隆接口。该路由已经有 `sanitizeName`，但它允许 Unicode 字母和数字通过，因此中文分身名仍会被作为 `customName` 传给上游。上游接口只接受 ASCII 字母、数字、`_` 和 `-`，并限制总长度不超过 64 个字符。

这个问题发生在服务端边界层，不涉及分身建模、记忆流或数据库写入。修复目标是让本地化分身名在不改 UI 展示名的前提下也能正常生成音色。

## 目标 / 非目标

**目标：**

- 让音色克隆请求始终生成满足上游约束的 `customName`
- 保证中文或混合字符分身名不会成为音色生成失败的根因
- 保持现有前端调用方式和接口返回结构不变

**非目标：**

- 不重做音色克隆交互
- 不新增数据库字段记录克隆名字
- 不改动分身展示名或要求用户输入额外别名
- 不处理与样本音频质量、转写质量无关的其他上游失败

## 决策

### 决策 1：只在服务端边界修正上游名字

- 选择：在 `/api/twin/voice/clone` 内统一生成合法 `customName`
- 原因：前端展示名允许本地化字符是合理产品行为，且客户端校验不能覆盖所有调用方。把兼容逻辑放在服务端边界最稳妥
- 备选方案：
  - 前端限制分身名只能输入 ASCII：会直接损伤产品命名自由度，放弃
  - 前端额外收集“音色英文名”：增加交互负担，放弃

### 决策 2：使用 ASCII slug + 长度裁剪 + 默认回退

- 选择：把名字转成仅包含 `[A-Za-z0-9_-]` 的 slug，压缩分隔符并裁剪到 64 个字符；若结果为空，则回退到安全默认值
- 原因：这是满足上游约束的最小改动，不需要引入转写或拼音依赖
- 备选方案：
  - 引入拼音或转写库把中文转成拉丁字母：增加依赖和不确定性，当前 bugfix 不需要
  - 直接固定写死一个名字：可行但丢失可读性，作为仅在 slug 为空时的回退

### 决策 3：保留现有错误链路，只移除“非法名字”这类可避免失败

- 选择：当请求仍失败时，继续读取并透传上游错误；仅确保名字不再触发已知校验错误
- 原因：这样不会掩盖音频、模型或网络问题，也便于继续定位非命名类失败
- 备选方案：
  - 统一吞掉所有上游错误并返回泛化提示：会降低排障能力，放弃

## 风险 / 权衡

- [名字回退后可读性降低] -> 仅在原始分身名没有任何可保留 ASCII 字符时启用默认值，优先保留可读 slug
- [固定长度裁剪可能截断过长名字] -> 上游本身有 64 字符限制，服务端裁剪是必需的；展示名仍保留完整原值
- [仍可能存在其他上游拒绝原因] -> 保留原有错误透传，避免把所有失败都误判为命名问题

## Migration Plan

- 无数据迁移
- 直接更新路由逻辑即可
- 如修复引发异常，可回滚该路由的命名处理逻辑，不影响已生成的 voice reference
