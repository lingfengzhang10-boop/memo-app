## ADDED Requirements

### 需求:长期资产必须具有生命周期状态
系统必须为长期资产提供生命周期状态，并根据确认、复用和过期情况进行状态转换。

#### 场景:一条新记忆被抽取并确认
- **当** 系统首次抽取一条长期资产候选
- **那么** 该资产必须先处于候选状态
- **那么** 用户确认后该资产必须进入已确认或稳定状态，而禁止继续与未确认状态混淆

### 需求:长期资产必须具有可见性和敏感性
系统必须为长期资产记录可见性和敏感性，以约束分身和其他消费面如何使用它。

#### 场景:分身准备引用一条长期资产
- **当** 分身准备主动提及某条长期资产
- **那么** 系统必须先检查该资产的可见性和敏感性
- **那么** 系统禁止主动暴露不允许主动提及的记忆

### 需求:修正必须保留版本链
系统必须保留长期资产的修正链或 supersede 关系，而禁止默认直接覆盖旧版本。

#### 场景:用户修正一条已确认记忆
- **当** 用户纠正或改写一条长期资产
- **那么** 系统必须保留新旧版本之间的关系
- **那么** 系统必须能够追溯每个版本的来源和确认时间

### 需求:冲突记忆允许共存
系统必须允许互相冲突的长期记忆在治理层共存，并通过状态、版本或优先级决定运行时采用哪一条。

#### 场景:同一主题出现前后不同说法
- **当** 用户在不同时间对同一主题给出不同描述
- **那么** 系统必须允许这些描述共存并保留来源
- **那么** 系统禁止在没有治理依据的情况下直接删去旧说法
