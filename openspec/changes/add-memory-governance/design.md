## 上下文

目前项目已经有“先确认再入库”的边界，但确认后的资产仍缺少统一治理语义。系统还不能稳定回答：

- 这条记忆是暂时的还是长期的
- 这条记忆能否主动被分身提起
- 这次修正是在纠错、改口还是重新理解
- 两条冲突记忆应该共存还是覆盖

没有治理层，后续检索越强，误用和越界风险越大。

## 目标 / 非目标

**目标：**

- 为长期资产统一定义生命周期与状态转移。
- 为长期资产统一定义可见性与敏感性。
- 保留修正和版本链，而不是直接覆盖旧理解。
- 让分身和其他消费面始终从受治理资产读取。

**非目标：**

- 不在此 change 中定义叙事层和意义层的抽取内容本体。
- 不在此 change 中设计新的前台地图或关系图页面。
- 不在此 change 中改变“一人一分身”边界。

## 决策

### 决策 1：所有长期资产都必须带生命周期状态

第一阶段统一采用：

- `candidate`
- `confirmed`
- `stable`
- `stale`
- `archived`

这样可以区分“刚猜到”“已确认”“长期成立”“可能过期”“只留档不参与日常”的内容。

### 决策 2：可见性和敏感性独立于生命周期存在

一条记忆可以是 `stable`，但仍然是 `private` 或 `sensitive`。生命周期解决“是否可信”，可见性解决“是否该说”。

### 决策 3：修正默认保留版本链

系统默认保留被修正前的版本和 supersede 关系，除非用户主动做删除。这样才能保留“人如何改变理解”的痕迹，也避免回忆录和传记失去解释链。

## 风险 / 权衡

- [风险] 状态和权限过多会增加实现复杂度。
  → 缓解措施：先统一语义与最小集合，不在第一阶段引入过细规则。

- [风险] 共存冲突记忆会让消费面逻辑更难。
  → 缓解措施：运行时始终优先 confirmed/stable 与最近确认版本，并保留冲突解释能力。

- [风险] 用户可能不理解为什么某条记忆“知道但不说”。
  → 缓解措施：在未来记忆工坊里显式展示状态和可见性。

## Migration Plan

1. 为主要长期资产补充治理语义。
2. 将确认流、编辑流、分身过滤逻辑统一切到治理语义上。
3. 在后续前台工坊与传记输出中复用这些字段。

## Open Questions

- 可见性是否拆成 `visibility` 与 `share_scope` 两套字段。
- 删除是否物理删除，还是一律转 `archived`。
